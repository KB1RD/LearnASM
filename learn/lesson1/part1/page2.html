My previous analogy, the one where a computer is a person holding a calculator, is somewhat inaccurate. The problem is that you can assume that a person has some general intelligence and can figure out how to use a calculator. That's just not true of computers. Computers are some of the dumbest machines on Earth that can add, if not the dumbest. Your set of instructions must have incredible detail. In my analogy, you would have to tell the person exactly which buttons to push. You can't get away with saying "Add one plus one." You would have to say: "Push one then push plus then push 1 then push equals then read the result."

Instructions

The way computers know which operation to do is through instructions. When we read and write instructions, they're made up of letters. By reading particular sequences of letters, we know what to do. Instructions in a computer are pretty similar. They're made up of particular sequences of 1s and 0s that the computer can understand. By reading these 1s and 0s, the computer knows what operations to perform on what data. Any slightly different sequence means that the computer may run a completely different operation.

The Assembler

Of course, I'm not expecting anyone to program in 1s and 0s. It may sound cool, but I speak from experience: It's just really annoying. So instead of going through that, we give a special program called the "assembler" text that it then turns into 1s and 0s for the computer. Don't worry! All of this is build right into LearnASM, so you don't need to worry about it.

Instructions vs. Programs

So... is my program an instruction?

Nope. With the exception of one small case, your program is not just one instruction, it's many instructions put together to do something useful. On its own, an instruction does virtually nothing. It's only when that instruction is put in context of the rest of the program that it makes any real sense. It's like asking if a word is a book. You could make a one word book, it'd just be a really boring book.

Your First Instruction

Speaking of instructions, it's time you learn your first instruction. Your first instruction is the following:

Name Function
halt Halts the program.

I suppose it's a bit of a disappointment, but still, it's a start.

You should see a "Halt condition detected" alert

Congradulations! You just wrote your first program! While it may not seem like a very complicated program, the "halt" instruction is critical. Try removing all the text and click run again. You should see an "Undefined Instruction" error. What this means is that the computer tried to read a section of memory with nothing in it. A normal computer wouldn't warn you about this. Instead, it could crash, do something completely random, or worse: it could run, say, a virus that just happens to be stored right after your program.

Why does this happen?

So your program is like a list. If the computer goes down this list that is your program and you haven't told it to stop, it doesn't know where the end is. So it'll just keep going. LearnASM will give you the "Undefined Instruction" error when the computer reaches the end of the list, but in future programs you may have some data stored at the end of the list like so:

  1. Code
  2. ...
  3. Code
  4. Data

But the computer, being really stupid, will interpret this data as an instruction and will end up doing something completely random. That's why you need halt: to tell the computer where the end is. The code above should become this:

  1. Code
  2. ...
  3. Code
  4. halt
  5. Data

Moral of the story: Put "halt" at the end of all your programs! Your program isn't done until you say so!
