Welcome to LearnASM!

Basics

LearnASM is a RISC load-store architecture. LearnASM is designed with the goal of being easy to learn and use relative to other assembly languages and for this reason it is not nearly as efficient as some other languages. LearnASM has 7 16-bit registers and a word size of 16 bits. The last three registers are reserved for special uses, but only the last register cannot be used as a general purpose register.

Register Name Alternate Name General Purpose? Functions
R0 Yes GP
R1 Yes GP
R2 Yes GP
R3 Yes GP
R4 Yes GP
R5 LR Yes GP. Used by the BL pseudocode. Also used by the deprecated link bit.
R6 SP Yes GP. Used by the PUSH and POP pseudocodes, but not treated differently by hardware.
R7 NI/PC No Stores the next address that will be executed.

CPU Setup

The CPU setup is very simple. The eight registers are placed along two input busses (SRC0 and SRC1) that feed into two inputs of the ALU. When the correct register is contained in the instruction, the value of a particular register is placed on the SRC0 bus, SRC1 bus, or both. The SRC1 bus is special: It also allows the input of an 8 bit immediate. The ALU interprets the opcode of the instruction and then determines what to do with the input data. During a MOV instruction, for example, it simply passes the SRC1 value along to the output. The output of the ALU then goes into the destination bus, which registers will accept as their value depending on the selected destination register in the instruction.

LDR and STR

LDR (load) and STR (store) instructions are accomplished by placing values from SRC0 and SRC1 onto the central bus and values from the central bus onto the destination bus. During these instructions, the ALU does nothing and writes neither 1s nor 0s to it's output, allowing the central bus to control the target register during a LDR instruction. During a STR instruction, no registers are impacted.

Flags

Flags are updated by setting the update bit in the instruction and will be updated by the ALU. During a LDR instruction, the Z (zero) and N (negative) flags are updated and all others are cleared. During a STR instruction, flags are not modified even if the update bit is set. Reading the flags is done using condition codes. Depending upon the value of the condition code bits in an instruction and the flags, an instruction may or may not execute. The condition code values will be covered in the Instruction Format section.

Instruction Format

The 32-bit instruction format is as follows:

Byte {{i-1}}
{{(i-1)%8}}
Cond. Code Opcode Mode U L Dest Src0 0 Src1 RESERVED
Cond. Code Opcode Mode U L Dest Src0 1 Src1 Immediate RESERVED

Condition Code

The condition code specifies whether the instruction will be executed or not depending upon the flags. Based on the result of a subtraction operation, it is possible to compare values, and so they are named based upon the results of such an operation. They are as follows:

Hex Name Condition Suffix(es)
0 Equal Z == 1 EQ
1 Not Equal Z == 0 NE
2 Greater than or equal to (unsigned) C == 1 CS/HS
3 Less than (unsigned) C == 0 CC/LO
4 Result is negative N == 1 MI
5 Result is positive or zero N == 0 PL
6 Overflow into sign bit V == 1 VS
7 No sign overflow V == 0 VC
8 Greater than (unsigned) (C == 1) && (Z == 0) HI
9 Less than or equal to (unsigned) (C == 0) || (Z == 1) LS
A Greater than or equal to (signed) N == V GE
B Less than (signed) N != V LT
C Greater than (signed) (Z == 0) && (N == V) GT
D Less than or equal to (signed) (Z == 1) || (N != V) LE
E Always 1 AL/none
F Never 0 NV

3 Bitwise XOR xor dest, src0, src1 Z and N modified, others cleared
4 Add add dest, src0, src1 C, V, Z, and N modified
5 Subtract sub dest, src0, src1 C, V, Z, and N modified
6 Unsigned Multiply umul dest, src0, src1 C, V, Z, and N modified
7 Signed Multiply smul dest, src0, src1 C, V, Z, and N modified
8 Unsigned Divide udiv dest, src0, src1 C, V, Z, and N modified
9 Signed Divide sdiv dest, src0, src1 C, V, Z, and N modified
A Logical Shift Left lsl dest, src0, src1 C, V, Z, and N modified
B Logical Shift Right lsr dest, src0, src1 C, V, Z, and N modified
C Arithmetic Shift Right asr dest, src0, src1 C, V, Z, and N modified
D Rotate Right ror dest, src0, src1 Z and N modified, others cleared

F Store str src1, src0 Flags are unchanged

For all instructions, (including LDR and STR) the first operand in the instruction is always the destination, or, in the case of STR, is the register that contains the address of the destination.

Mode

The mode bits determine which parts of registers are used and are as follows:

Hex Name Source Register Usage Destination Register Usage Suffix
0 None All 16 bits None N
1 Top Upper 8 bits Upper 8 bits T
2 Bottom Lower 8 bits Lower 8 bits B
3 Full All 16 bits All 16 bits F

The use of the "None" mode is to allow compare instructions to be performed without impacting registers.

Update (U) Bit

The update bit is very simple: If it is set, the ALU output will modify the flags.

Link (L) Bit Deprecated

The purpose of the link bit was to provide single-instruction branching with linking, however, its use seems to be very limited and for that reason it is deprecated and should not be used until a replacement is decided upon. When set, the link bit will copy the original contents of the destination register into the link register (R5) regardless of the mode bits.

Destination, Src0, and Src1

The destination and src0 registers both work the same: the register number is stored in the three bit space in the instruction. The src1 value may be from a register or it may be from an immediate stored in the instruction. The source of the src1 value is changed by the value of bit 18. If bit 18 is zero, the next three bits will be used as a register ID just like a destination or src0 register. If bit 18 is one, the next eight bits will be used as an immediate value.

The Virtual Machine

The LearnASM Virtual Machine currently has little functionality other than the CPU functionality mentioned above. There are only RAM and ROM banks in the system, however, there is space for memory-mapped IO registers in the future. The memory map is as follows:

Start End Function Length
0x0000 0x6FFF A general purpose RAM bank. No special functionality. 28 kB
0x7000 0x7FFF Reserved for future memory-mapped IO. Writes are discarded and reads have undefined results. 4 kB
0x8000 0xFFFF A general purpose ROM bank. Any assembled code is placed here. 32 kB

Reset Vectors

Upon powerup, the first thing the CPU does is read the address 0xFFFE-0xFFFF for the reset vector. The CPU will immediately branch to this reset vector.

Assembly & Directives

Opcode Interpretation

Instructions must be in a certain format for the assembler to understand what the user is attempting to write. Generally, the format is as follows:

opcode + Mode Suffix + Link & Update Suffixes + Condition Codes destination, source, source
For example, if I wanted to write an "add lower bits on equal with update" instruction, I could write the following:
addbueq r0, r1, #5

Labels

Labels are currently only used as targets in branching. A label is created by placing a colon after text containing no white space like so:

my_label:

Labels always take the address of the next instruction or byte assembled, so a label at the very end of a program does nothing.

Immediates

Take my previous example:

addbueq r0, r1, #5

That #5 was an immediate value. As mentioned before, immediates are only available for a src1 value, which is always at the end of an opcode and is used by every instruction except STR. An immediate value is indicated by placing a # symbol in front of a number. In addition, hexadecimal values can be used and are indicated by placing a 0x in front of a hexadecimal value. However, since the immediate values are only 8 bits, an immediate may not be greater than 255. If an immediate value greater than 255 is required, the mode bits can be used like so:

movb r0, #0x34
movt r0, #0x12

...which will result in R0 having a value of 0x1234. When using a branch instruction, immediates can also be a label. They are indicated by placing < and > signs around the label name like so:

b <my_label>

Pseudocodes

With the current instruction set, there are a number of "standard" features that aren't supported by hardware, so they are implemented by pseudocodes that code for other instructions.

(add lr, pc, #2)

Directives

Directives are indicated by placing a . before the directive name. A list of directives is as follows:

Name Arguments Function
.origin Origin Address The origin directive tells the assembler where the start of the image is located in memory. This defaults to 0x8000, which is the location of the 32kB ROM block of memory. The origin directive should only be used once in a program.
.seek Seek Address The seek directive tells the assembler to move to a particular address and place the following instructions and data there. The address is relative to the origin, so if the origin is set correctly, it should be the real location in memory.
.location Seek Address The location directive is an alias for the seek directive.
.byte value Places a single byte in the output image.
.word value Places a single 16-bit word in the output image.

Comments

Comments are indicated by placing a semicolon at the end of a line. Anything on the line after the comment will not be treated as code. A comment is written like so:

mov r0, r0 ; Wastes time

Implicit Reset Vectors

By default, the origin is set to 0x8000 (you should never need to change this) and the reset vector is set at the same address. This way, a user can just start typing and forget about reset vectors. You really shouldn't ever need them, but they're there in case you'd like to see how they would work in a real machine. The equivalent of the defaults is as follows:

start:
; my code here

.seek 0xFFFE
.word <start>

Future Changes

Since the deprecation of the link bit, I have been thinking about the following instruction format:

Byte {{i-1}}
{{(i-1)%8}}
Cond. Code Opcode Dest U Src0 0 Src1 Mode RESERVED
Cond. Code Opcode Dest U Src0 1 Src1 Immediate

My only concern is that it becomes annoying to increment half-register 8-bit counters. (like with addb r0, r0, #1) Furthermore, a student could write "addb r0, r0, #n" and wonder why on earth that won't work, thus creating a weird rule that is annoying and difficult to work around. A possible solution could be the following:

Byte {{i-1}}
{{(i-1)%8}}
Cond. Code Opcode Dest U Src0 0 0 Src1 RESERVED Mode RESERVED
Cond. Code Opcode Dest U Src0 0 1 Src1 8-bit Immediate Mode RESERVED
Cond. Code Opcode Dest U Src0 1 Src1 Immediate

But that's awfully complicated.
